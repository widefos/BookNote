### 程序的局部性原理

程序的局部性原理(principle of locality)是常用的程序属性。它是指程序执行时所
访问的存储器地址分布不是随机的,而是相对簇聚。现在常用的一个经验规则是,程序
执行时间的90%都是在执行程序中10%的代码。数据访问也具有局部性,不过其局部
性弱于代码访问的局部性。
局部性包括时间局部性和空间局部性。时间局部性是指程序即将用到的信息很可
能就是目前正在使用的信息。空间局部性是指程序即将用到的信息很可能与目前正在
使用的信息在空间上相邻或者临近。




## 指令集

### RISC（精简指令集）

尽可能将指令简化，指令条数少，指令功能比较简单

### CISC（复杂指令集）

将越来越多的功能交给硬件实现，以增加处理器复杂度为代价提升性能

缺点：

- 各种指令使用频率相差悬殊
- 指令集庞大，指令条数多，指令复杂，使得控制器硬件复杂
- 许多指令操作繁琐，CPI的值大，执行速度慢
- 指令功能复杂，规整性不好不利用采用流水技术来提高性能



- ARM使用RISC
- X86使用CISC

#### 五段流水线

- IF取指：从指令寄存器中读出指令，并将PC+4
- ID译码：对指令进行译码，根据指令内容找到并从通用寄存器中读出操作数
- EX执行：对上一个周期传入的操作数进行运算或处理
- MEM访存：从内存中读取或者写入数据
- WB写回：将运算结果和从内存中读取的数据写入寄存器


#### 相关和冲突解决

**相关类型（相关指的是指令之间存在依赖关系）**：

- 数据相关
- 名相关
- 控制相关

数据相关是后面的指令所处的阶段要使用前面指令某一阶段所得到的结果

名相关，名指的是寄存器或者存储单元名称，有两种情况
- 反相关：一个指令要写入，一个指令要读出，并   且都是同一个寄存器
- 输出相关：两条指令都要写同一个寄存器或者内存地址

控制相关：
主要在分支指令当中
![[Pasted image 20250618154511.png]]

**冲突类型（冲突指的是，由于有某种相关的存在，使得指令不能在特定的时钟周期执行）**：

- 结构冲突
- 数据冲突
- 控制冲突

结构冲突是由于流水线当中硬件资源不足而引发的，例如如果某个流水线将指令和数据放在一起存储，那么如果一条指令处在访存阶段而另外一条指令处于取指阶段，那么就会发生冲突

>解决办法是将后一条指令延迟一个或者多个指令周期，避开冲突阶段



数据冲突由数据相关引发，有以下几种情况

1. 指令j需要指令i的数据，但是指令i还没有写回
2. 指令j在指令i写入同一存储单元之前就进行了写入操作，导致写入顺序错误
3. 指令j在i读取之前就先进行了写入操作，导致i读取到错误数据

>解决方法：定向技术或者采用停顿延迟指令周期来执行


定向技术就是在任何时候都将EX或者MEM段的流水寄存器的结果回传给EX入口处，这样就解决了拿不到数据或者


控制冲突由条件语句引发，例如下面情况就是：

j ...
add ..
sub ...

第一个是跳转语句，当读入第一条语句的时候因为是流水处理器同时也读入了后两条语句，如果跳转成立那么后两条语句就无效了，这就是控制相关

> 解决方法是当读到控制语句时，暂停控制语句之后语句的执行，也就是让流水寄存器暂停写入，并暂停IF阶段的取指，直到控制语句完成对于PC值的修改，对于流水寄存器来说是在MEM阶段完成。
> **总结解决方法就是读到控制语句暂停下一条指令执行，等待控制语句结果，如果成功，则让IF段内PC跳转为目标地址，如果失败则让PC继续执行下一条指令**







### 向量处理方式

![[Pasted image 20250617193134.png]]

总共有三种处理方式

- 横向处理
- 纵向处理
- 纵横处理方式

**横向处理**：
就例子来说，处理过程是

先计算 $d_1 \leftarrow  a_1*(b_1+c_1)$
...
然后 以此类推计算出  $d_n \leftarrow  a_n*(b_n+c_n)$

这就是横向处理的方式，对每个分量进行单独计算


**纵向处理方式**：

计算  $k_1 \leftarrow（b_1+c_1）$
....
$k_n \leftarrow（b_n+c_n）$

再计算 $d_i \leftarrow k_i*a_i$

每条指令都只负责一种运算

#### 向量处理机冲突及处理

**源向量冲突**：
`V3 <- V1 + V2` 和 `V5 <- V4 ∧ V1`
两条指令都需要访问V1，这就造成了冲突，只能等一条指令执行完毕释放V1才能执行下一条

**功能部件冲突**：
`V3 <- V1 * V2` 和 `V5 <- V4 * V6`
两个运算都需要使用乘法部件，而同一功能部件同时只能执行一个运算，这也会冲突，必须等待一条指令执行完成相应部件才能空出执行下一条


#### 提高处理机性能的方式

1. 设置多个功能部件
2. 链接技术

>**链接技术**：

**核心目标：** 加速处理**具有数据依赖（先写后读相关）** 的多个向量指令序列，让它们像一条更长的流水线一样工作，而不是完全串行执行。


**前提条件（必须满足）：**

1. **数据相关：** 指令序列必须是 `指令A -> 指令B -> ...` 的形式，即 `指令B` 的某个**源向量寄存器**恰好是 `指令A` 的**结果向量寄存器**。
    
2. **无资源冲突：**
    
    - **无功能部件冲突：** `指令A` 和 `指令B` 使用的**功能部件（加法器、乘法器等）必须是不同的**。不能都抢同一个计算单元。
        
    - **无源向量冲突：** `指令A` 和 `指令B` 除了那个相关的寄存器外，**不能有其他的源向量寄存器是同一个**（避免前面解释过的“抢仓库”问题）。


**操作方式**：

这其实就是向量版的，旁路转发，之前的标量流水器为了解决先写后读问题，当运算结果得出的时候在写会之前直接就将结果给到ID模块，这里也是类似的，在上一条指令的运算结果写回寄存器之前，就将结果送到运算部件，这里针对的是向量中的每一个元素。

- 从宏观上看，`指令B` **不必等待 `指令A` 处理完整个 `V3` 向量**（即等所有元素都算完并存进 `V3`）才开始。它只需要**等 `指令A` 算出第一个元素**就可以开始了。（指令A的每个元素得出结果都会被直接送到运算部件中）
    
- 后续元素的处理在两条指令的功能部件间紧密衔接，如同一条流水线。




## 存储系统



存储系统层次取决于三个关键要素：**容量**，**速度**，**价格**，三个要素必定不能被同时满足，就像一个不可能三角一样

- 速度越快，价格越高
- 容量越大，价格越低
- 容量越大，速度越慢

为了解决这三个要素所产生的矛盾，所以我们将存储系统按层次进行设计，不同的层次对应不同的功能，从而合理搭配这三要素，发挥最佳性能


**程序访问局部性原理**：

- 时间局部性：程序很可能马上将要用到的信息可能是现在正在使用的信息
- 空间局部性：程序很可能马上将要用到的信息与现在的信息在存储上是相邻的

基于这个局部性原理，我们就有了三层设计结构：




### 存储性能

这就是存储器1的命中率，N1,N2分别代表每个存储器的访问次数

#### 映像规则

**全相连映象**：主存当中的任一块都可以被调入Cache当中相关的位置。

- 特点：空间利用率最高，冲突概率最小，实现复杂

**直接映像**：主存中的一块只能被放入Cache当中的唯一的一块位置

- 特点：空间利用率最低，冲突概率最高，实现最为简单

**组相连映像**：主存中的每一块可以被放置到Cache中唯一的
一个组中的任何一个位置。

- 特点：组相联是直接映像和全相联的一种折中

### 查找方式


查找解决如何用主存地址确定该地址是否在Cache内，以及具体在哪


**索引目录**：为了记录Cache当中的那一块具体对应哪个主存块，就需要一个索引目录，记录数和Caceh当中的块数对应

- 每一行记录包含两个关键信息：
    
    - **标识：** 记录着**当前存放在这个 Cache 块里的数据，它原本属于哪个主存块**。简单说，就是贴在这个 Cache 块上的那个“身份证号”（Tag）。
        
    - **有效位：** 一个简单的标志（0 或 1）。`1` 表示：“我这个 Cache 块里现在装的东西是有效的、有用的数据”。`0` 则表示：“我这个块是空的，或者里面的数据是无效的垃圾数据（比如刚开机还没用，或者被覆盖了）”。



主存地址：Cache的控制器会将主存地址解析问三部分
![[Pasted image 20250619165200.png]]

- 标识：主存块的唯一标识，所有块唯一
- 索引：表明这个块具体对应 Cache当中的哪个块
- 块内位移：块内的具体位置


具体查找步骤：


1. CPU 说：“我要内存地址 `X` 的数据！”
    
2. **拆解地址：** Cache 控制器瞬间把地址 `X` 拆成三部分：`标识_X`，`索引_X`，`位移_X`。
    
3. **定位候选位置：** 用 `索引_X` 直接找到 Cache 目录表中对应的**那一行或那几行**记录（候选位置）。这步非常快，因为索引直接指向物理位置，就像直接翻到电话本的第几页。
    
    - _如果是直接映射_：索引指向**唯一**的一行记录（一个候选位置）。
        
    - _如果是组相联映射_：索引指向一个**组**（几行记录，几个候选位置）。
        
    - _如果是全相联映射_：索引没用，整个目录表都是候选（效率最低，现实中很少用）。
        
4. **核对“身份证”：** Cache 控制器拿出 `标识_X`，去和它找到的候选位置目录表记录里的 `标识` 字段**逐个比较**。
    
5. **检查“有效性”：** 同时检查这些候选位置记录里的 `有效位` 是不是 `1`。
    
6. **判断结果：**
    
    - **命中：** 如果在某个候选位置的记录里，`标识` 和 `标识_X` **完全匹配**，并且该记录的 `有效位` 是 `1`。恭喜！这意味着 CPU 要的数据就在这个 Cache 块里！
        
        - 控制器根据 `索引_X` 找到对应的 **Cache 块**。
            
        - 再用 `位移_X` 从这个块里**精确取出** CPU 要的那个字节/字。
            
        - 数据飞速送给 CPU。
            
    - **未命中：** 如果所有候选位置的 `标识` 都不匹配，或者匹配的那个 `有效位` 是 `0`。抱歉，Cache 里没有 CPU 要的数据。
        
        - 控制器必须启动慢得多的操作：去主内存里把包含地址 `X` 的**整个主存块**加载到 Cache 的一个块里（通常选一个候选位置）。
            
        - 加载时，会把这块的目录表记录更新：`标识` 字段设为 `标识_X`，`有效位` 设为 `1`。
            
        - 然后才能把 CPU 要的具体数据（由 `位移_X` 指定）送给 CPU。



### 替换算法

替换算法解决，如果Cache没命中需要从主存中读取数据并将存储块放入Cache中时该如何替换Cache的块


- 随机替换算法（RAND）
- 先进先出（FIFO）
- 近期最少使用（LRU）


#### LRU算法硬件实现


**堆栈法**：
以组相连为例，每个组内都设有一个栈，这个栈记录组内每个块访问的先后次序。
- 栈底记录最先被访问的块
- 当组内满的时候需要替换就需要栈底的元素块

特点：速度低，成本较高（适用于关联度较小的）


**比较对法**：

将组内的块两两进行比较得到一个相对次序。

### 写策略

当需要修改内存当中的内容时，因为Cache比内存快，所以修改的时候优先修改Cache当中的值，这就造成主存和Cache当中数据不一致的问题，因为Cache相当于是主存的一个副本，所以如何保证数据的一致性就是写策略要解决的问题

主要有两种类型的方法：

1. 写直达法：同时更新主存和Cache当中的值，当写入数据命中Cache的时候，就同时修改Cache和主存当中的值，如果没有命中就先将主存当中的块加载到Cache中，然后再进行写入
2. 写回法：写入的时候只修改Cache当中的内容，然后把这块做个标记，表明已经被修改过，之后如果需要替换这块的数据再把数据写入主存当中。


### Cache性能分析

$$平均访存时间=命中时间+不命中率+不命中开销$$

$$CPU时间=(CPU执行周期数+存储器停顿周期数)*时钟周期时间$$

$$存储器停顿时钟周期数=访存次数×不命中率×不命中开销$$
不命中开销指的是未命中时所花费的周期数

### 通道处理机

>概念：管理各种外设的处理器

**功能**：
- 接收CPU的IO指令并连接相应外设
- 执行通道程序
- 给出外设中要进行读写操作所在的地址
- 给出主缓冲区的首地址
- 指定传送工作结束时要进行的操作


### 通道种类

- 字节多路通道：每次只传输一个字节，传完换一台设备继续传
- 选择通道：每次选择一台设备传输，传完这台设备的数据后再传下一台
- 数组多路通道：每次传输k个字节，传完换一台设备继续传


### 通道流量分析

>通道流量指的是，单位时间内通过通道的数据量

基本参数概念：
- 设备选择时间：从通道第一次收到设备的数据传送请求开始吗，第一次传送数据的时间
- 传送一个字节所需要的时间
- 连接到通道上的数据数
- 每台设备传送的字节数：每台设备需要传送多少字节
- **数组多路通道传输的数据块大小 (字节数)**：在“数组多路通道”这种工作方式下，通道每次为一台设备连续传输多少字节后，就切换到下一台设备。`k` 就是这个连续传输的“数据块”大小。
- **通道完成全部数据传送工作所需的总时间**： 从通道开始响应第一个设备的请求，到为所有 `p` 台设备都完成它们 `n` 字节数据传输所花费的总时间。
