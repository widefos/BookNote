三类等待：

等待的作用：自动化过程中，页面访问的时候，需要一定的时间来进行加载、页面加载资源需要时间，所以需要程序进行等待。


Selenium 主要提供了三种等待方式：**隐式等待**、**显式等待**和**强制等待**。

#### 1. 隐式等待

- **概念**：为整个 WebDriver 的生命周期设置一个全局的等待时间。当 WebDriver 在 DOM 中查找一个或多个元素（如果立即可用则立即返回）时，如果元素没有立即找到，WebDriver 会轮询 DOM 一段时间，直到找到该元素或超时。
    
- **工作方式**：设置一次，对整个驱动程序实例（Driver）都有效，直到你再次更改它。
    
- **语法**：
    ```
    
    from selenium import webdriver
    
    driver = webdriver.Chrome()
    # 设置隐式等待时间为10秒
    driver.implicitly_wait(10)
    
    # 后续的所有 find_element 操作都会受到这个隐式等待的影响
    driver.get("https://example.com")
    element = driver.find_element("id", "dynamic-content")
    ```
- **优点**：
    
    - 设置简单，一行代码即可。
        
    - 对脚本中所有的 `find_element` 和 `find_elements` 方法都生效。
        
- **缺点**：
    
    - 不够灵活，无法针对某个特定条件进行等待。
        
    - 可能会因为等待最大时间而降低整体测试效率（比如一个元素本来1秒就出现了，但它还是会等到10秒超时吗？**不会**，它一找到就会继续执行）。
        
    - 无法处理除“元素存在”以外的其他条件（如元素可点击、元素可见等）。
        

#### 2. 显式等待

- **概念**：针对某个特定的条件和元素进行等待。你可以设置一个最大超时时间，以及一个检查条件是否满足的频率。WebDriver 会反复检查这个条件，直到条件为真（返回一个非空值）或超时。
    
- **核心组件**：
    
    - `WebDriverWait`：等待类。
        
    - `expected_conditions`：预期条件集合，提供了大量预定义的条件。
        
- **语法**：
    
    ```

    
    
    from selenium import webdriver
    from selenium.webdriver.common.by import By
    from selenium.webdriver.support.ui import WebDriverWait
    from selenium.webdriver.support import expected_conditions as EC
    
    driver = webdriver.Chrome()
    driver.get("https://example.com")
    
    # 创建一个 WebDriverWait 实例，设置最大等待时间为10秒
    wait = WebDriverWait(driver, 10)
    
    # 使用 until 方法，传入一个预期条件
    # 这个条件的意思是：等待直到 ID 为 ‘dynamic-button’ 的元素可以被点击
    element = wait.until(EC.element_to_be_clickable((By.ID, "dynamic-button")))
    element.click()
    ```
- **常用预期条件**：
    
    - `presence_of_element_located`：元素出现在 DOM 中（不一定可见）。
        
    - `visibility_of_element_located`：元素不仅出现在 DOM 中，而且可见。
        
    - `element_to_be_clickable`：元素可见且可被点击。
        
    - `text_to_be_present_in_element`：元素的文本包含特定文字。
        
    - `alert_is_present`：等待警报框出现。
        
- **优点**：
    
    - **非常灵活**，可以精确控制等待的条件。
        
    - **高效**，条件一旦满足就立即继续执行，不会浪费不必要的等待时间。
        
    - 功能强大，能处理各种复杂的异步加载场景。
        
- **缺点**：
    
    - 代码比隐式等待更复杂，需要为每个需要等待的操作单独编写。
        

#### 3. 强制等待

- **概念**：使用编程语言的休眠函数（如 Python 的 `time.sleep()`）来让脚本暂停执行一段固定的时间。
    
- **语法**：
    ```
    import time
    from selenium import webdriver
    
    driver = webdriver.Chrome()
    driver.get("https://example.com")
    
    # 无论页面状态如何，强制等待5秒
    time.sleep(5)
    
    # 5秒后再执行后续操作
    element = driver.find_element("id", "some-element")
    ```
- **优点**：
    
    - 简单粗暴，易于理解。
        
- **缺点**：
    
    - **极其不推荐**在正式脚本中使用。
        
    - **效率极低**：如果设置时间过短，元素可能还没加载出来；如果设置时间过长，会造成大量不必要的时间浪费。
        
    - 不可靠，无法适应网络或服务器响应速度的变化。
        

---

### 最佳实践与组合使用

1. **首选显式等待**：在绝大多数情况下，**显式等待是首选和推荐的方式**。它提供了灵活性、可靠性和高效性。
    
2. **谨慎使用隐式等待**：
    
    - 可以设置一个相对较短的隐式等待（例如 5-10 秒）作为全局后备，处理那些简单的、稳定的元素查找。
        
    - **注意**：隐式等待和显式等待**不要混用**，尤其是在同一个查找操作上，因为这可能导致不可预测的总等待时间。官方文档也不推荐混用。如果要用显式等待，最好将隐式等待设置为 0。
        
3. **避免使用强制等待**：除非在极少数特殊场景下（例如等待一个非网页的副作用完成，且没有更好的检测方法），否则永远不要使用 `time.sleep()`。
    
4. **组合使用示例**：
    
```
from selenium import webdriver
    from selenium.webdriver.common.by import By
    from selenium.webdriver.support.ui import WebDriverWait
    from selenium.webdriver.support import expected_conditions as EC
    
    driver = webdriver.Chrome()
    # 设置一个较短的隐式等待作为全局后备
    driver.implicitly_wait(5)
    
    try:
        driver.get("https://example.com")
        
        # 对于关键操作，使用显式等待，并覆盖隐式等待
        # 将隐式等待临时设为0，防止与显式等待冲突
        driver.implicitly_wait(0)
        wait = WebDriverWait(driver, 15) # 显式等待15秒
        dynamic_element = wait.until(
            EC.visibility_of_element_located((By.ID, "ajax-loaded-content"))
        )
        print("动态内容加载成功！")
        # 恢复隐式等待
        driver.implicitly_wait(5)
        
        # 其他常规操作由隐式等待处理
        some_button = driver.find_element(By.NAME, "submit")
        some_button.click()
        
    finally:
    driver.quit()

```


### 补充：网页页面加载原理

### 把网页加载比作 **"在餐厅点餐和上菜"**

想象一下你去一家餐厅（访问一个网站）吃饭。

---

### **1. 点第一道菜：请求HTML（骨架）**

你告诉服务员："我要一份套餐。"（浏览器向服务器说："给我这个网页。"）

服务员很快端上来一个**空餐盘和一张菜单**。这就像浏览器收到了最基本的 **HTML 文件**。

- **空餐盘** = 网页的空白区域
    
- **纸质菜单** = **HTML 骨架**。它上面印着菜的**名字**和**位置**，比如"左上角是开胃菜"，"中间是主菜"，但菜本身还没上来。它只定义了网页的结构和哪里该放什么东西。
    

这时你看菜单，你知道这里_应该_有一道菜，但你还吃不到。

---

### **2. 根据菜单下单：加载CSS（装修和摆盘）**

接下来，餐厅会根据菜单上的描述，开始**布置餐桌**。

- 给你换上漂亮的桌布、精致的刀叉、摆上一束花。
    
- 这就是 **CSS** 文件的作用。它负责**美化页面**，定义颜色、字体、布局、大小。没有CSS，网页就是一堆光秃秃的文字和链接堆在一起。
    

现在，餐桌看起来像样了，但你依然没有吃到真正的食物。

---

### **3. 开始上菜：加载静态资源（上主菜）**

现在，真正的"菜"开始端上来了，填补菜单上预留的空位。

- **图片**：就像服务员端上来一盘漂亮的牛排图片。这就是加载 `JPG`, `PNG` 等图片文件。
    
- **字体**：就像给菜单换上了更优雅的艺术字。
    
- **基础的JavaScript**：可能像是一个自动旋转的展示台，让你的甜盘慢慢转动，方便你观赏。
    

到这一步，一个**传统的、静态的网页**就算加载完成了。你看到了所有内容，也可以"吃"（看）了。

---

### **4. 特色菜和现做菜：动态加载（AJAX/JavaScript）**

但这家餐厅很高级，有些菜是**现点现做**的。

你看到菜单上有一道"**今日主厨推荐**"，旁边有一个"**点击获取**"的按钮。你按下了这个按钮。

这时，服务员（**JavaScript**）并没有给你换一张新菜单，而是**悄悄地**跑到后厨（**服务器**），问主厨今天的推荐菜是什么。然后他很快跑回来，**只把这一道新菜的名字和描述，贴在了菜单的空白处**。

**这个过程就是 AJAX 或动态加载。**

- **整个页面没有刷新**（你没换桌子，也没换菜单）。
    
- **只有一小部分内容被更新了**（只是菜单上多了一行新菜）。
    

现代网页（如社交媒体、邮箱、项目管理工具）充满了这种操作：

- 滚动页面到底部，自动加载更多内容。
    
- 点击"发送"按钮，只是把你写的这条新消息添加到对话列表中，而不会刷新整个页面。
    

---

### **总结与Selenium等待的关系**

现在，回到Selenium为什么需要等待：

1. **如果你不等**：菜单（HTML）刚拿来，你就急着用手指着菜单上一个"牛排图片"的位置说："我要吃这个！"但服务员会告诉你："先生，图片还没端上来呢，您指的位置现在什么都没有。" —— 这就是 `NoSuchElementException`。
    
2. **隐式等待**：你规定一个时间，比如1分钟。在这1分钟内，你每隔几秒就指一下那个位置问："有了吗？" 一旦菜上来了，你就不问了。如果1分钟还没上来，你才发脾气（报错）。
    
3. **显式等待**：你更聪明。你不对所有菜都用同一个策略。对于那道需要现做的"主厨推荐"，你设定了**特定的等待条件**。你会一直等，直到：
    
    - **条件A（元素可见）**："我能清清楚楚看到菜名了。"
        
    - **条件B（元素可点击）**："那个'下单'按钮从灰色变成红色，我可以点击了。"
        
    - 只要条件满足，你立刻行动，一点也不浪费时间。
        
4. **强制等待**：你不管菜有没有上来，直接闭上眼睛睡5分钟。醒来后，不管三七二十一，就直接去指那个位置。这种方法最傻，要么等太久浪费时间，要么醒来发现菜还是没上来。
    

所以，理解页面是**分阶段、动态**加载的，就能明白为什么自动化脚本必须"耐心"地等待正确的元素在正确的状态下出现，才能进行成功的操作。
