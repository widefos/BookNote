
1.权限问题，权限是怎么实现的





Spring Security 是 Spring 生态中用于**认证、授权和安全防护**的强大框架。

**权限控制**

- **基于 URL**：对不同接口（如 `/admin/**`）限制访问角色。
    
- **基于方法**：在 Service 或 Controller 方法上使用 `@PreAuthorize("hasRole('ADMIN')")` 注解。
    
- **动态权限**：根据用户自己的数据（如只能查看自己发的帖子）实现细粒度控制。


有关@PreAuthorize("hasRole('ADMIN')")` 注解的原理：

数据库中存放着角色表，如下面所示：

```
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    username VARCHAR(50),
    password VARCHAR(100)
);

-- 角色表（注意角色名通常存 'ROLE_ADMIN' 或 'ADMIN'）
CREATE TABLE roles (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50)  -- 内容如：'ADMIN', 'USER', 'MANAGER'
);

-- 用户-角色关联表
CREATE TABLE user_roles (
    user_id BIGINT,
    role_id BIGINT,
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (role_id) REFERENCES roles(id)
);
```

角色表和用户表相关联，

之后需要实现 `UserDetailsService`类，在加载用户时把数据库中的角色转换成 Spring Security 能识别的 `GrantedAuthority`：



当你在类或者接口上加上注解 `@PreAuthorize`的时候：

```
@PreAuthorize("hasRole('ADMIN')")
public void deleteAllUsers() { ... }
```

框架会去`GrantedAuthority` 集合中自动查找是否有匹配的ROLE_ADMIN（角色码）

缺陷状态流转机制：


- 在 DefectServiceImpl 里维护三张规则表（本质上是map映射集合<String ,set<String>>）：
- ACTION_FROM_STATUS：某动作允许从哪些状态执行
- ACTION_TO_STATUS：该动作执行后变成什么状态
- ACTION_ROLE_CODES：哪些角色能执行该动作
- 例如：
- FIXING --(FIX)--> FIXED
- FIXED --(VERIFY_PASS)--> CLOSED
- FIXED --(VERIFY_FAIL)--> REOPENED


每次缺陷改变状态通过查询这三张表就能得到下一次的状态
